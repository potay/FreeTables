What is this in one line?

CAS-based lock-free list (CAS is compare and swap) with lock-free hash table upon it.

What is this?

We will implement a compare and swap based lock-free linked list that supports insert and delete and will build a hash-table that supports insert, delete and lookup upon it.

The algorithms for implementing the following can be found here.

What should I know to understand this project?

In parallel programs, multiple threads need to read and modify shared data structures. One way to ensure that the programs updates is correct is to allow the updates to the shared data structures to happen atomically. This involves introducing locking mechanisms in the shared data structures such as global locks and fine grain locks.

However, adding locks increase contention. Contention slows down the performance of parallel programs. At high-thread counts, lock free data structures can perform better than even data-structures with fine-grained locks. Thus the motivation for lock-free data-structures (more info).

We wish to implement a lock-free linked list that can be used as a basis to develop a lock free hash table. The hash-table is a commonly used data structure. It is used in a wide variety of applications that require constant time lookup. A correct, efficient lock-free hash-table can significantly speed up parallel programs.

In fact, if we still have time we will extend our implementation to implement a parallel binary decision package. Binary decision diagrams have applications to verification, AI planning, and combinatorial optimization and the commonly known packages are sequential.

How is this challenging

Our workload will consists of a sequence of inserts and deletes into first the linked list and later once we have implemented a hash table inserts, deletes and lookups into the hash table. We need to come up with a workload that deals with conditions when there are a large number of inserts relative to deletes and vice versa.

We also need to measure generate workload that is suitable to hashing.

  • Correctness: Correctness is our major challenges. Multiple threads will be attempting to insert and delete into the linked list and insert/delete into the hash-table. We have to make sure that the threads correctly update the state of the data-structure so that we do not lose elements, dereference null pointers etc. We also have to make sure that the memory is managed properly and that all allocated data-structures are freed.

  • Efficiency: Efficiency is another one of our concerns. We will be developing for the Xeon Phi and have to ensure that all twenty four execution contexts can operate on the shared data-structure without significantly slowing down the performance. For example once we get the hash table working, we may have false sharing that could occur when threads from processors (each processor has a separate L3) make requests to data that hashes to the same location.

What are our goals and deliverables?

What do we plan to achieve?
  1. Implement a correct, efficient CAS-based lock-free list that supports insertion and deletion.
  2. Test and measure its performance versus a list implemented using single grained and fine-grained locks for varying thread counts.
  3. Implement a correct, efficient lock-free working hash table that supports insertion, deletion and lookup.
  4. Test and measure its performance versus a hash table implemented using single grained and fine-grained locks varying thread counts.
What do we hope to achieve?
  1. Parallelize a binary decision diagram package on the Phi. Only if project goes spectacularly well though.
